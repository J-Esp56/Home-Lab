## 📌 Summary

Performed an active reconnaissance and vulnerability assessment of my home network (`192.168.1.0/24`) using Kali Linux. Enumerated all connected devices, identified open ports and services, analyzed traffic passively via Wireshark, and assessed security posture per device. Identified two medium-risk findings and documented remediation steps.

## 🔧 Steps Taken

1. Ran initial Nmap scan across full subnet to discover live hosts and open ports:
    ```bash
    nmap -T4 -A 192.168.1.0/24
    ```

2. Investigated smart outlet — initial scan flagged CVE-2010-0738 (JBoss JMX) and CVE-2010-2333 (LiteSpeed). Ran aggressive version scan to confirm actual service:
    ```bash
    nmap -sV -A -p 80 192.168.1.22
    ```
    Confirmed SHIP 2.0 — all CVE flags were false positives.

3. Ran passive traffic capture on smart outlet via Wireshark. Observed only gratuitous ARP broadcasts — no outbound traffic. Confirmed SHIP 2.0 uses mutual TLS, requiring certificate exchange before communication.

4. Probed Eufy Homebase 3 RTSP service:
    ```bash
    curl -v rtsp://192.168.1.50/live --max-time 10 -X DESCRIBE -H "Accept: application/sdp"
    ffmpeg -i rtsp://192.168.1.50/live0 -vframes 1 test.jpg
    ```
    RTSP server active but stream gated behind Eufy app toggle. No unauthenticated access confirmed.

5. Probed EufyCam 3 port 8888 with curl and netcat:
    ```bash
    curl -v http://192.168.1.199:8888
    nc -v 192.168.1.199 8888
    ```
    Both connected successfully but received no response — device waiting for proprietary binary handshake. Identified as Eufy's local API protocol via model hostname `T8600`.  Source: bropat (github)

6. SSH tested Unifi APs — connected successfully confirming SSH is open:
    ```bash
    ssh ubnt@192.168.1.10
    ```
    Default credentials (`ubnt/ubnt`) rejected. Password auth still enabled — key-based auth confirmed as local credentials

7. Tested ADB access on Fire TV:
    ```bash
    adb connect 192.168.1.200:5555
    adb -s 192.168.1.200:5555 shell
    ```
    Connection attempted — returned `failed to authenticate`. Token auth correctly rejected unapproved device.

8. Fingerprinted second Amazon device via HTTP headers:
    ```bash
    nmap -sV -p 6000,8008,8009,8080 --script http-headers 192.168.1.225
    ```
    Server header `AirReceiver/1.0.3.0` confirmed Fire TV running AirReceiver casting app. No ADB exposed.

9. Ran UDP scan across IoT devices:
    ```bash
    sudo nmap -sU --top-ports 100 192.168.1.22
    ```
    All 100 ports ignored — silently dropped by firewall.

## 🚨 Findings / Errors

- **Nmap false positives** → Smart outlet initially flagged for JBoss and LiteSpeed CVEs. Aggressive version scan confirmed SHIP 2.0 — both flags were script misidentification due to unusual 200 OK behavior on any requests.

- **Unifi AP SSH password auth enabled** → SSH is required for controller communication and cannot be disabled. However password authentication is active 

- **Fire TV ADB exposed on LAN** → Port 5555 open and accessible from any device on the network. Token auth is working and rejected unapproved connection. Risk exists if a previously approved token is compromised or if device is moved to an unsegmented network.

- **Eufy port 53 TCP open** → Unusual for a camera hub. Returns `tcpwrapped`. Likely proprietary Eufy communication channel using port 53 to bypass firewall restrictions — not functioning as a DNS server.
